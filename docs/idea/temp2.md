# v0 Pro 작업 범위 분할안 - kds-web frontend

## 전체 원칙

이번 작업은 `kds-web` 프론트 UI 개선 작업이다.

백엔드 API, DB, Auth 로직, store context 로직은 새로 만들지 않는다.
기존 API와 상태 구조를 최대한 유지하고, 필요한 경우 프론트 local state 또는 mock state로 UI 구조만 먼저 잡는다.

반드시 유지할 것:

```text
Auth / token 로직
store context
GET /api/kds/orders
PATCH /api/orders/{order_id}/status
진행중 / 완료 탭 구조
주문 카드 완료 버튼
기존 주문 데이터 렌더링 구조
```

---

# 작업 1차: 메인 KDS 주문 카드 핵심 UX 개선

## 목적

메인 KDS의 `진행중` 탭에서 주문 카드 내부 메뉴를 클릭하면 완료 / 미완료 상태가 배경색으로 토글되도록 한다.

또한 신규 주문 표시 방향, 스크롤 UX, 신규 주문 강조, 새로고침 버튼의 기본 인터랙션을 함께 정리한다.

## 작업 범위

### 1. 메뉴 완료 / 미완료 토글

진행중 탭의 주문 카드 내부 메뉴 라인을 클릭 가능하게 만든다.

동작:

```text
메뉴 클릭
→ 해당 메뉴 완료 상태로 변경
→ 메뉴 라인 배경색이 연한 초록색 계열로 변경

완료된 메뉴 다시 클릭
→ 미완료 상태로 변경
→ 기본 배경으로 복귀
```

완료 상태는 텍스트를 추가하지 않고 시각적 변화로만 표현한다.

권장 표현:

```text
미완료 메뉴:
기본 흰색 배경

완료 메뉴:
연한 초록색 배경
메뉴명 색상도 초록 계열로 변경 가능
```

주의:

```text
메뉴 옆에 "완료", "완료 1", 게이지, 공정 라벨 같은 텍스트를 추가하지 않는다.
```

메인 KDS는 1초 안에 주문을 파악해야 하므로 메뉴명, 수량, 옵션 가독성을 해치지 않는다.

---

### 2. 옵션 처리 정책

메뉴에 옵션이 딸려 있는 경우, 메뉴 완료 처리는 해당 메뉴와 그 메뉴에 포함된 옵션까지 함께 완료 처리하는 것으로 본다.

예시:

```text
부대찌개 x2
ㄴ 공기밥 추가
ㄴ 라면 사리 추가
```

`부대찌개` 메뉴 라인을 클릭하면 다음 전체 블록이 완료 상태로 표현된다.

```text
부대찌개 x2
ㄴ 공기밥 추가
ㄴ 라면 사리 추가
```

즉 옵션을 별도로 클릭하거나 별도 완료 상태로 관리하지 않는다.

정책:

```text
메뉴 클릭 = 메뉴 + 해당 메뉴 옵션 전체 완료/미완료 토글
옵션 단독 클릭 없음
옵션 단독 완료 상태 없음
옵션은 항상 부모 메뉴의 완료 상태를 따라간다.
```

완료 상태가 되었을 때는 메뉴명뿐 아니라 옵션 영역도 같은 완료 배경 안에 포함되도록 한다.

---

### 3. 주문 도착 순서 변경

신규 주문은 기존과 반대로 **최우측에 추가**한다.

이유:

```text
왼쪽 → 오른쪽으로 시간 흐름이 자연스럽다.
오래된 주문이 왼쪽에 남아 우선 처리하기 쉽다.
신규 주문이 오른쪽에 추가되어 흐름이 직관적이다.
```

---

### 4. 스크롤 개선

주문 보드는 가로 스크롤을 유지한다.

단, 하단 스크롤바는 시각적으로 숨긴다.

목표:

```text
스크롤 가능
스크롤바는 보이지 않음
터치패드 / 마우스 휠 / 드래그로 탐색
PC에서는 마우스 상하 스크롤을 가로 스크롤로 치환
```

---

### 5. 신규 주문 강조 효과

주문이 새로 도착하면 해당 카드에 일시적인 outer glow 또는 breathe 효과를 적용한다.

```text
신규 주문 도착
→ 카드 외곽 glow
→ 일정 시간 후 사라짐
```

주방에서 새 주문을 즉시 인지하기 위한 시각 피드백이다.

---

### 6. 새로고침 버튼 개선

새로고침 버튼은 실제 주문 목록을 갱신해야 한다.

동작:

```text
새로고침 클릭
→ 아이콘 한 바퀴 회전
→ 주문 영역 갱신
→ 주문 카드 fade-out / fade-in
```

이미 새로고침 API 호출이 있다면 기존 로직을 사용한다.
없다면 현재 주문 목록을 다시 fetch하는 기존 함수 또는 local refresh 흐름으로 연결한다.

---

```text
메뉴 완료 시 해당 메뉴의 옵션 영역도 함께 완료 배경에 포함된다.

기존 API/Auth/store context 로직은 깨지지 않는다.
```

# 작업 2차: 주문 카드 액션 / 완료 탭 정리 / 카드 배치 로직

## 목적

주문 카드의 보조 액션과 완료 탭 정리 기능을 추가한다.

또한 주문 카드의 길이에 따른 배치 로직을 정리한다.

## 작업 범위

### 1. 주문 카드 롱클릭 / 우클릭 액션

주문 카드를 길게 클릭하거나 우클릭하면 액션 팝업을 띄운다.

선택지:

```text
상세정보
제거
```

### 상세정보

주문 상세정보를 보여준다.

포함 정보:

```text
주문번호
주문 시간
메뉴
옵션
요청사항
결제금액
플랫폼
주소
연락처
```

주소와 연락처는 민감정보이므로 기본 카드에는 표시하지 않고 상세정보 팝업에서만 보여준다.

### 제거

주문 제거 확인 모달을 띄운다.

```text
주문을 제거하시겠습니까?
[예] [아니오]
```

초기에는 실제 주문 취소가 아니라 KDS 화면에서 숨김 또는 제거 처리로 정의한다.

현재 백엔드 API가 없다면 프론트 상태에서만 숨김 처리한다.

---

### 2. 완료 탭 주문 내역 정리

`완료` 탭에서는 새로고침 아이콘 옆에 휴지통 아이콘을 추가한다.

동작:

```text
휴지통 클릭
→ 주문완료 내역을 삭제할까요?
→ 예 / 아니오
```

목적:

```text
완료된 주문이 계속 쌓여 화면이 복잡해지는 문제 해결
영업 중 KDS 화면 정리
```

초기 구현에서는 hard delete보다 archive 또는 hidden 처리에 가깝게 가져가는 것이 안전하다.

백엔드 API가 없다면 프론트 상태에서 완료 탭 목록만 비우는 방식으로 UI를 먼저 구현한다.

---

### 3. 주문 카드 배치 로직

#### 짧은 카드 2행 배치

다음 조건을 만족하면 주문 카드 2개를 한 컬럼 안에 2행으로 배치할 수 있다.

```text
1. 현재 주문 카드가 짧다.
2. 다음 주문 카드도 짧다.
3. 두 주문을 2행으로 배치해도 내용이 잘리지 않는다.
```

목적:

```text
짧은 주문들이 화면을 과하게 차지하지 않게 함
한 화면에 더 많은 주문 표시
```

#### 긴 카드 2열 메뉴 배치

하나의 주문 카드가 길 경우 메뉴 목록을 2열로 배치할 수 있다.

조건:

```text
메뉴 개수가 많음
카드 세로 길이가 과하게 길어짐
2열로 나누어도 가독성이 유지됨
```

주의:

```text
2열 배치로 인해 메뉴와 옵션의 관계가 헷갈리면 안 된다.
옵션은 반드시 부모 메뉴 아래에 붙어서 표시되어야 한다.
```

---

```text
상세정보 팝업에서 민감정보를 확인할 수 있다.
제거 클릭 시 확인 모달이 뜬다.
완료 탭에 휴지통 아이콘이 추가된다.
```

# 작업 3차: 운영 상태 인디케이터 / 통계 탭 / 설정 탭 UI

## 목적

매장 운영에 필요한 상단 운영 상태, 통계 탭, 설정 탭의 프론트 UI를 추가한다.

이번 작업은 UI 중심이다.
실제 API 연동이나 DB 저장은 하지 않는다.

## 작업 범위

### 1. 운영 상태 인디케이터

KDS 상단에 운영 상태 인디케이터를 추가한다.

상태:

```text
영업중
일시중지
영업종료
```

인디케이터를 누르면 상태 변경 팝업을 연다.

### 영업중

정상 주문 수신 상태.

### 일시중지

일정 시간 동안 주문 수신을 중지한다.

일시중지 시간 설정:

```text
기본값: 10분
- 버튼: 10분 감소
+ 버튼: 10분 증가
```

### 영업종료

당일 주문 수신 종료 상태.

현재 API가 없다면 local state로만 상태 변경 UI를 구현한다.

---

### 2. 통계 탭

`통계` 탭을 추가한다.

초기 표시 항목:

```text
오늘 메뉴별 주문 수
오늘 총 매출
오늘 완료 주문 수
```

후속 확장 가능한 항목:

```text
일간 / 주간 / 월간 매출
기간별 주문 수
시간대별 주문량
인기 메뉴
```

통계 탭은 실시간 주문 처리 화면과 분리해서, 매장주가 영업 흐름을 확인하는 화면으로 둔다.

현재 API가 없다면 mock data 또는 기존 주문 데이터 기반 계산으로 UI만 구성한다.

---

### 3. 설정 탭

`설정` 탭을 추가한다.

설정 항목:

```text
알림 on/off
알림 사운드 설정
브레이크타임 설정
비밀번호 변경
주문 자동수락 on/off
```

### 알림 on/off

주문 도착 알림을 켜거나 끈다.

### 알림 사운드 설정

주문 도착 시 재생할 사운드를 선택한다.

### 브레이크타임 설정

특정 시간이 되면 자동으로 일시중지 상태로 전환한다.

예시:

```text
브레이크타임 시작: 15:00
중지 시간: 60분
```

### 비밀번호 변경

비밀번호 변경 성공 시 현재 세션을 로그아웃한다.

프론트 처리:

```text
accessToken 삭제
refreshToken 삭제
AuthPage로 이동
```

이번 작업에서는 실제 비밀번호 변경 API가 없으면 UI만 만든다.

### 주문 자동수락 on/off

```text
ON:
주문 수신 즉시 진행중 표시

OFF:
주문 수락 버튼을 누른 뒤 진행중 표시
```

현재 API가 없으면 설정값은 local state로 처리한다.

---

## 제외 범위

이번 3차 작업에서는 하지 않는다.

```text

직원 관리 페이지 구현
내 업무 탭 구현
```

## 완료 기준

```text
상단에 운영 상태 인디케이터가 보인다.
운영 상태 변경 팝업이 열린다.
일시중지 시간 선택 UI가 있다.
통계 탭이 추가된다.
통계 탭에 오늘 메뉴별 주문 수, 총 매출, 완료 주문 수가 표시된다.
설정 탭이 추가된다.
알림, 사운드, 브레이크타임, 비밀번호 변경, 자동수락 UI가 표시된다.
기존 KDS 주문 처리 화면은 깨지지 않는다.
```

# 작업 4차: 직원 계정 [내 업무] 탭 UI

## 목적

직원 계정에서 사용할 `[내 업무]` 탭 UI를 만든다.

`내 업무` 탭은 완료 처리 화면이 아니다.
직원이 자신이 등록한 담당 메뉴와 현재 남은 업무량을 확인하는 화면이다.

완료 처리는 메인 KDS의 `진행중` 탭에서만 한다.

## 화면 구조

직원 계정의 상단 탭은 다음 형태로 표시한다.

```text
[진행중] [완료] [내 업무]
```

관리자/매장주 계정의 직원 관리 페이지 이름은 `[직원]`이고, 직원 계정의 업무 확인 탭 이름은 `[내 업무]`다.

이번 작업에서는 실제 권한 분기 API를 새로 만들지 말고, 기존 로그인 사용자 정보나 임시 조건으로 UI 구조만 반영한다.

---

## 내 업무 탭 구성

`내 업무` 탭은 두 영역으로 구성한다.

```text
1. 담당 메뉴 요약 카드 영역
2. 업무 히스토리 영역
```

---

### 1. 담당 메뉴 요약 카드 영역

상단에는 직원이 등록한 담당 메뉴별 남은 수량을 카드 형태로 표시한다.

예시:

```text
[짜장면]        [짜장면 곱빼기]    [짬뽕]        [짬뽕곱빼기]
   3                 3              0              1
```

카드 구성:

```text
메뉴명
큰 숫자
```

표시 규칙:

```text
카드 하나는 담당 메뉴 하나를 의미한다.
숫자는 현재 진행중 주문 중 아직 메인 KDS에서 완료 처리되지 않은 해당 메뉴의 총 수량이다.
수량이 0이어도 카드는 유지한다.
수량이 0인 카드는 비활성 느낌으로 표시해도 된다.
```

카드 스타일:

```text
메뉴명은 상단 또는 중앙 상단
수량은 크게 표시
카드 간 간격은 넉넉하게 유지
수량이 많은 메뉴가 시각적으로 바로 들어오도록 숫자를 크게 표시
```

---

### 2. 업무 히스토리 영역

하단에는 현재 직원 담당 메뉴와 관련된 주문 항목을 테이블 형태로 표시한다.

컬럼:

```text
주문번호
메뉴
갯수
시각
상태
```

예시:

```text
| 주문번호 | 메뉴       | 갯수 | 시각               | 상태   |
|---------|------------|------|--------------------|--------|
| ABCD13  | 짜장면     | 1    | 6월 17일 오후 7:43 | 진행중 |
| ABCD12  | 짬뽕곱빼기 | 1    | 6월 17일 오후 7:35 | 진행중 |
| ABCD12  | 짜장면     | 2    | 6월 17일 오후 7:35 | 진행중 |
| ABCD11  | 짬뽕       | 4    | 6월 17일 오후 7:23 | 완료   |
```

업무 히스토리는 직원이 자신의 업무가 어떤 주문에서 발생했는지 확인하기 위한 영역이다.

---

## 담당 메뉴 관리 UI

`내 업무` 탭 상단 또는 카드 영역 근처에 `[메뉴 추가]` 버튼을 둔다.

필수 기능 UI:

```text
담당 메뉴 추가
담당 메뉴 수정
담당 메뉴 삭제
```

동작은 실제 API 연동 없이 프론트 상태로만 구현해도 된다.

메뉴 추가:

```text
[메뉴 추가] 클릭
→ 메뉴명 입력
→ 저장
```

메뉴 수정:

```text
담당 메뉴 카드 또는 관리 버튼 클릭
→ 메뉴명 수정
→ 저장
```

메뉴 삭제:

```text
삭제 클릭
→ "이 담당 메뉴를 삭제하시겠습니까?"
→ 예 / 아니오
```

---

## 메인 KDS와의 연동 표현

현재 프론트 상태 안에서 메인 KDS 메뉴 완료 상태가 있다면, `내 업무` 탭의 수량과 상태가 자동으로 바뀌도록 한다.

예:

```text
짜장면 3
```

메인 KDS에서 짜장면 1개가 완료 처리되면:

```text
짜장면 2
```

메인 KDS에서 메뉴 완료 상태를 해제하면 `내 업무` 탭에도 다시 진행중으로 반영된다.

---

## 금지사항

`내 업무` 탭에는 완료 버튼을 두지 않는다.

하지 말 것:

```text
내 업무 탭에서 메뉴 완료 처리
완료 숫자 표시
원형 게이지
공정 표시
직원별 완료 기록 UI
```

## 완료 기준

```text
직원 계정 화면에 [진행중] [완료] [내 업무] 탭이 보인다.
내 업무 탭 상단에 담당 메뉴별 수량 카드가 보인다.
내 업무 탭 하단에 업무 히스토리 테이블이 보인다.
담당 메뉴 추가/수정/삭제 UI가 있다.
내 업무 탭에서는 완료 처리를 하지 않는다.
메인 KDS 메뉴 완료 상태에 따라 내 업무 수량과 상태가 갱신된다.
```

# 작업 5차: 직원 관리 페이지 / Auth 전환 애니메이션 / 최종 폴리싱

## 목적

관리자/매장주 계정의 직원 관리 페이지와 Auth 페이지 전환 애니메이션을 추가하고, 전체 UI 명칭과 스타일을 정리한다.

## 작업 범위

### 1. 직원 관리 페이지

`kds-web`에는 매장주 또는 관리자 계정용 직원 관리 페이지를 추가한다.

관리자 계정에서 직원 관리 페이지는 네비게이션에 `직원`으로 표시한다.

직원 관리 페이지 기능:

```text
직원 목록
직원 추가
직원 PIN 발급
직원 비활성화
직원 상태 확인
```

이번 작업에서는 실제 API 연동 없이 UI 구조만 먼저 만든다.

직원 관리 페이지는 매장주가 직원을 생성하고 관리하는 화면이다.

직원 계정에서 보이는 `내 업무` 탭과는 역할이 다르다.

---

### 2. Auth 페이지 전환 애니메이션

로그인 / 매장가입 탭 전환 시 fade 효과를 적용한다.

```text
로그인 → 매장가입
매장가입 → 로그인
```

효과:

```text
fade-out
fade-in
```

회원가입 플로우의 완성도를 높인다.

---

### 3. 역할별 탭 명칭 정리

관리자/매장주 계정:

```text
[진행중] [완료] [통계] [설정] [직원]
```

직원 계정:

```text
[진행중] [완료] [내 업무]
```

`직원`과 `내 업무`를 혼용하지 않는다.

---

### 4. 최종 UI 폴리싱

정리할 것:

```text
주문 카드 가독성
메뉴 완료 배경색
옵션 들여쓰기
상단 탭 간격
내 업무 카드 간격
업무 히스토리 테이블 가독성
모달 버튼 스타일
```
